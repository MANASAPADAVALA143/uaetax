## Cursor execution playbook

1. **One prompt per session.** Paste a single numbered prompt, complete it, test, commit. Start a fresh composer for the next prompt so context stays under ~20k tokens.

2. **Standard prefix** (paste before each prompt body):

   > Context: see `docs/BUILD_PLAN.md` for overall architecture. Work only on the scope described below. Do not refactor unrelated files. After changes, run tests and report back.

3. **Before accepting a change**, ask the agent: *What files did you change and why?* and *What is the regression risk to VAT / CT / dashboard?*

4. **Prompts 5 (Supabase auth) and 6 (CT module)** — use **Plan Mode** first (multi-file, security-sensitive). Prompts 1–4 are usually fine in agent mode.

5. **Git hygiene:** one branch per prompt, e.g. `prompt-1-bulk-classify-contract`; merge to `main` only after manual smoke tests.

6. **Anthropic model strings:** use **`claude-opus-4-7`** (or the current Opus generation your org standardizes on). Do not let templates drift to `claude-opus-4-6`, `claude-3-5-sonnet`, or other stale IDs.

7. **n8n + app:** E-invoicing automation lives in `n8n/workflows/UAE_EInvoicing_Readiness.json` (generated by `scripts/generate_einvoicing_n8n.py`). Setup steps: `docs/einvoicing_setup_guide.md`. App-side parity is **Prompt 7**; inbound automation is **Prompt 11** (HMAC webhook).

## Product snapshot (mid-market UAE)

- **ICP:** AED 10M–500M revenue, finance-led, B2B-heavy; **hero hook:** e-invoicing (Peppol PINT AE, Phase 1 from Jan 2027 for large taxpayers, ASP appointment pressure).
- **Stack:** FastAPI backend, Next.js frontend, RAG, optional n8n for scheduled / notification orchestration.
- **Positioning:** unified compliance surface (VAT + CT + deadlines + e-invoicing readiness), not “VAT classifier only.”

## Prompt 2 — DB verification (when Postgres is running)

### Bring up Postgres (Docker)

Requires [Docker Desktop](https://www.docker.com/products/docker-desktop/) (or Docker Engine) on your PATH.

```bash
docker run -d --name gulftax-db \
  -e POSTGRES_DB=gulftax \
  -e POSTGRES_USER=gulftax \
  -e POSTGRES_PASSWORD=gulftax \
  -p 5432:5432 postgres:15
```

Copy `backend/.env.example` → `backend/.env` (same `DATABASE_URL` works for the container above).

### Migrations

From `backend/` with `DATABASE_URL` set (or loaded from `.env` via your shell):

```bash
alembic upgrade head
```

Confirm revision `002_verify_dashboard` is applied, then check the column (PostgreSQL):

```sql
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'transactions'
  AND column_name IN ('verification_history', 'transaction_type');
```

Codebase: there should be **no** `from_orm(` usage (Pydantic v2 uses `model_validate`). Search with `rg "from_orm"`.

## E-invoicing workflow (n8n) — design note

The importable workflow uses **13 nodes**: schedule → config (split: sheet IDs + API constants) → Google Sheets read → **Calculate E-Invoicing Readiness** (Code) → **Filter** `Status = READY` → **single Claude HTTP** (`claude-opus-4-7`) → compile → Sheets append + Gmail + Slack + **disabled** GulfTax webhook.

A literal **IF → two Claude nodes → Merge** pattern can deadlock in n8n when only one branch runs; this template uses one advisory call with engine-built context. You can refactor to dual Claude + routing once you are comfortable with n8n execution semantics.

## After you change the workflow

```bash

```

Then re-import or diff in n8n. Edit the generator script, not the JSON by hand, when possible.
